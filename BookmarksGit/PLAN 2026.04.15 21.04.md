# Canonical Title Mapping Plan

## Summary
Adopt **mod title IDs as the canonical namespace** for all `b_ / c_ / d_ / k_ / e_ / h_` mapping work. Build a single editable master mapping that relates every imported vanilla title to the mod namespace, then drive safe rewrites from that table into `history` and other title-consuming systems.

This is the right default because the current repo already references mod IDs heavily in `history/titles`, `geographical_regions`, `on_action`, `scripted_effects`, and localization for titles such as `k_balhae`, `k_shiwei`, `k_longxi_china`, `d_shanzhou`, and `c_hezhou_china`.

## Implementation Changes
- Create a source-of-truth mapping file at `Works/map_data_sources/title_relation_master.csv`.
- Make `mod_title_id` canonical. Each row represents one vanilla title and must include:
  - `vanilla_title_id`
  - `vanilla_tier`
  - `mod_title_id`
  - `mod_tier`
  - `relation_type`
  - `rewrite_allowed`
  - `source_root_title`
  - `notes`
  - `status`
- Use these mapping semantics:
  - `relation_type = exact` for true title equivalents.
  - `relation_type = contextual` for titles that are not literal renames but are the same gameplay/historical slot in this mod.
  - `status = manual_review` when no defensible `exact/contextual` match exists yet.
- Do **not** add a runtime alias layer. This stays a build-time/reference-time mapping system.
- Do **not** force a nearest-match rewrite. Automatic rewrites are allowed only when the title is a direct or defensible contextual counterpart.
- Every vanilla title under managed imported areas must appear in the master table.
  - If it has no safe counterpart yet, it still gets a row with `status = manual_review`, and you review it manually later.
- Seed the table from existing provenance data already produced by tooling:
  - `source_root_title / source_kingdom / source_duchy / source_county / source_barony`
  - province-based evidence from current old/new/RGB mapping outputs
  - current live references already present in `history`, `geographical_regions`, and script files

## Consumer Rules
- Add a generator/validator toolchain that consumes `title_relation_master.csv` and emits:
  - a safe rewrite map for title references
  - a manual-review report for unresolved rows
  - a coverage report showing which vanilla titles are mapped, rewriteable, or blocked
- Apply automatic rewrites only to rows where:
  - `status = mapped`
  - `rewrite_allowed = yes`
  - `relation_type IN (exact, contextual)`
- Use the mapping in these systems first:
  - `history/titles`
  - `history/provinces`
  - `map_data/geographical_regions`
  - title references in `common/scripted_effects`, `common/on_action`, and related script files
- Leave localization keyed to the canonical mod IDs. No global localization migration is required unless a canonical mod title itself lacks a loc key.

## Test Plan
- Coverage test: every imported vanilla `b_/c_/d_/k_/e_/h_` title has a row in `title_relation_master.csv`.
- Safety test: only `exact/contextual + mapped` rows are emitted into rewrite outputs.
- Manual-review test: unresolved titles are reported explicitly and never silently rewritten.
- Regression test on representative titles:
  - `k_balhae`
  - `k_shiwei`
  - `k_longxi_china`
  - `d_shanzhou`
  - `c_hezhou_china`
- Post-rewrite acceptance:
  - no increase in `Title does not exist`
  - no new `Province has no associated title in common/landed_titles`
  - no duplicate province assignments introduced by mapping-driven changes

## Assumptions and Defaults
- Canonical namespace: **mod title IDs**
- Rewrite policy: **only direct or contextual equivalents**
- No-match policy: **manual review**, not forced nearest mapping
